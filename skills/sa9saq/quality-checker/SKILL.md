# 品質チェックスキル

納品前の品質保証を自動化するスキル。

## 概要

### なぜ必要か
```
目的:
├── 納品物の品質を一定に保つ
├── ミスを事前に発見
├── クレームを防止
├── 信頼を築く
└── 修正回数を減らす
```

---

## チェック対象

### コード系
```
GAS/JavaScript/Python:
├── 構文エラーがないか
├── 動作確認済みか
├── コメントが適切か
├── 変数名がわかりやすいか
├── エラーハンドリングがあるか
├── 機密情報がハードコードされていないか
└── 不要なconsole.logが残っていないか
```

### スプレッドシート系
```
├── 数式が正しく動くか
├── 書式が統一されているか
├── 印刷範囲が適切か
├── データ検証が設定されているか
└── サンプルデータが入っているか
```

### ドキュメント系
```
├── 誤字脱字がないか
├── 説明がわかりやすいか
├── スクリーンショットが最新か
├── 手順が抜けていないか
└── FAQ/トラブルシューティングがあるか
```

---

## チェックリスト

### 納品前必須チェック
```
□ 要件を満たしているか
□ 動作確認済みか
□ エラーが出ないか
□ マニュアルが添付されているか
□ ファイル形式は正しいか
□ 機密情報が含まれていないか
□ クライアント名/固有情報が残っていないか
□ プロフェッショナルな見た目か
```

### カテゴリ別追加チェック

**GAS案件**
```
□ トリガーの設定方法が説明されているか
□ 権限の説明があるか
□ 外部APIのキー設定方法があるか
```

**スプレッドシート案件**
```
□ 使い方の説明シートがあるか
□ 保護設定が適切か
□ モバイル表示を確認したか
```

---

## 自動チェック機能

### コード検証
```
自動で実行:
├── Lintチェック（ESLint/Pylint）
├── 構文チェック
├── 未使用変数の検出
├── セキュリティパターンの検出
│   ├── APIキーのハードコード
│   ├── eval()の使用
│   └── SQLインジェクションリスク
└── 未完了マーカーの残存チェック
```

### ドキュメント検証
```
自動で実行:
├── スペルチェック
├── リンク切れチェック
├── 画像の存在確認
└── 文字化けチェック
```

---

## 品質スコア

### スコアリング
```
100点満点で評価:
├── 機能性: 40点（要件を満たすか）
├── 完成度: 30点（エラーなし、動作OK）
├── ドキュメント: 20点（マニュアルの質）
└── 見た目: 10点（プロフェッショナルか）

合格ライン: 80点以上
```

---

## 不合格時の対応

### 自動修正
```
軽微な問題は自動修正:
├── コードフォーマット
├── 明らかな誤字
├── 不要な空白/改行
└── コメントの形式統一
```

### 手動修正が必要な場合
```
アラート表示:
├── 問題箇所を明示
├── 修正方法を提案
├── 優先度を表示
└── 納品可否を判断
```
